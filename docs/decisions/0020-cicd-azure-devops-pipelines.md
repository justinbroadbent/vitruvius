# ADR 0020 — CI/CD architecture: OIDC-federated, plan-gated, with a generated deployment ledger

## Context

[ADR 0007](./0007-change-as-code.md) sets the change-as-code controls — the pull request is the change record, **CD** (continuous delivery, the automation that performs deployments) is the executor, a deployment ledger is the audit artifact, and drift is detected in **CI** (continuous integration, the automation that checks every proposed change) — and keeps those controls platform-agnostic. This ADR specifies the pipeline architecture that carries them: how the deploy identity authenticates, where plan and apply run, what the ledger contains, and how drift is caught.

The architecture is platform-agnostic. The reference implementation is Azure DevOps Pipelines triggered from a GitHub source repository; an adopter on a different platform maps the same controls onto it through ADR 0007's portability table.

## Decision

### 1. Plan is a pull-request check; apply is gated, approved, and promoted per environment

- `plan` — Terraform's dry run, which shows what would change without changing anything — runs as a PR check. No apply runs on a PR.
- The plan is also the input to the **conformance check** ([ADR 0025](./0025-deployment-conformance-and-platform-baseline.md)): the deployment's declared profile is evaluated against the rendered plan, and the PR cannot merge if a rule fails without a recorded exemption ([ADR 0008](./0008-audit-before-deny-policy-lifecycle.md)).
- `apply` — the step that makes real changes — runs after merge, per environment, behind an approval gate. A production apply requires an approver who is not the author. Apply consumes the exact plan that was reviewed and conformance-checked — bound by the plan hash recorded in the ledger (§3), never a silent re-plan — so what ships is what was proven.
- Promotion is dev → staging → prod, each its own environment-subscription ([ADR 0024](./0024-landing-zone-binding-and-scope-vocabulary.md)). The same built artifact promotes through every environment; environments differ by input, not by rebuild.
- Standard / normal / emergency changes ([ADR 0007](./0007-change-as-code.md) §4) map to conditional stages and PR tags.

### 2. The deploy identity is OIDC workload-identity federation

The pipeline authenticates to Azure through a **workload-identity-federation (OIDC)** connection — the pipeline proves who it is with a short-lived token Azure already trusts, instead of holding any stored credential. So: no PATs (personal access tokens), no service-principal secrets, no storage-account keys ([ADR 0009](./0009-secrets-ephemeral-by-default.md)). The same federated identity holds least-privilege access to the Terraform state backend ([ADR 0017](./0017-terraform-state-and-backend.md)) and scoped write access to the target environment-subscription ([ADR 0024](./0024-landing-zone-binding-and-scope-vocabulary.md)).

### 3. The deployment ledger is generated by the apply stage

Each deployment emits an append-only **ledger** record — a log entry that can be added to but never edited: the PR link, commit SHA, plan/artifact hash, target environment and scope, executor identity, approver, timestamp, and outcome. The ledger is generated from pipeline state — never maintained by hand — and is durable and queryable. It serves double duty: the audit record, and the source of the DORA delivery signals ([ADR 0013](./0013-platform-metrics-and-dora.md)).

### 4. A scheduled pipeline detects drift

A scheduled `plan` runs against each root's state; if it finds any difference between code and reality (a non-zero plan), it opens a work item ([AP-004](../anti-patterns.md#ap-004--configuration-drift), [ADR 0007](./0007-change-as-code.md) §6).

### 5. Pipeline agents run with a least-privilege identity and a path to private endpoints

State and many other resources are network-private ([ADR 0017](./0017-terraform-state-and-backend.md) §2, [ADR 0018](./0018-network-topology-hub-spoke.md)). The agents that run apply and the drift check therefore need network reachability to the platform's private endpoints, and they run as the federated identity (§2), not as a broadly-privileged account.

### 6. CI validates the module contract

PR validation runs `fmt` / `validate` / `terraform test`, plus manifest validation: each `manifest.yaml` is validated against `schemas/module-manifest.schema.json` and checked for parity with the module's `variables.tf` / `outputs.tf` ([ADR 0011](./0011-module-manifest.md)).

> **In plain terms:** the pipeline never holds a password, it shows its work on every proposed change (the plan) and checks that plan is compliant before it can merge, it needs a second person's sign-off to touch production, and it writes its own tamper-evident receipt for every deployment.

## What this does not decide

- **The CI/CD platform.** Azure DevOps Pipelines is the reference implementation; the controls are platform-agnostic. An adopter's platform is their choice.
- **Concrete org, project, agent-pool, and service-connection identifiers, and the agent network topology** — adopter and networking-layer concerns.
- **The ledger's storage and field schema** — a durable, queryable store; the record's contents are fixed here.
- **Standard-change pattern definitions and the commit-signing mechanism** — set by [ADR 0007](./0007-change-as-code.md).
- **The break-glass diff-capture tooling** — the requirement (a captured back-fill PR within 24 hours) is [ADR 0007](./0007-change-as-code.md) §5.

## Reversibility

The controls are load-bearing; the platform under them is not. Porting to another CI/CD platform means mapping each control to its equivalent (ADR 0007's table) — configuration work, not architecture work. Two parts must hold from the start. The deployment-ledger contract (§3): the ledger feeds DORA ([ADR 0013](./0013-platform-metrics-and-dora.md)) and the audit story, so once anything consumes it, its fields can only be added to, never removed or changed. And the OIDC-only identity posture (§2): letting static secrets back in would dismantle the no-stored-credentials evidence ([AP-006](../anti-patterns.md#ap-006--secret-rotation-toil)).

## Consequences

**Positive.**

- No static secrets anywhere in CD: one federated identity authenticates to Azure, to state, and to the target subscription (§2).
- One generated artifact — the ledger — is both the audit evidence and the DORA signal source (§3).
- Segregation of duties and environment promotion are enforced by gates, not ceremony ([AP-007](../anti-patterns.md#ap-007--change-management-theater)).
- Plans are visible on every PR; applies are approved and promoted.

**Negative — and accepted.**

- A split source/CD toolchain means two control planes to wire — branch protection on the repo side, pipeline gates on the deploy side. The controls hold across both ([ADR 0007](./0007-change-as-code.md)).
- Agents that can reach private endpoints are more setup than hosted runners — the cost of network-restricted state and resources ([ADR 0017](./0017-terraform-state-and-backend.md), [ADR 0018](./0018-network-topology-hub-spoke.md)).
- The ledger and manifest-validation steps are build effort, paid once.
